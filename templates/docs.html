1. Présentation & Contexte Général

Ymmo est un groupe immobilier d'envergure nationale implanté en France, composé d'un siège social basé à Aix-en-Provence et d'un maillage territorial de 12 agences réparties dans les grandes métropoles.

Afin de répondre à la fois aux besoins de ses clients (recherche, acquisition et estimation en ligne) et de ses agents (supervision commerciale, suivi de la performance et gestion de la relation prospects), Ymmo a développé sa plateforme web centralisée intégrant des outils d'analyse statistique et de prédiction de prix (moteur IA).

🎯 Objectifs Académiques B2 (Ynov Informatique) :

  • Développement Avancé : Élaboration d'un backend orienté services, typé et modulaire en Python.
  • Bonnes pratiques & POO : Structure logicielle respectant les patterns SOLID, DRY et KISS.
  • SQL Avancé : Modèle relationnel robuste avec jointures, indexations, agrégations et intégrité référentielle stricte.
  • Data & IA : Algorithme prédictif d'apprentissage supervisé en Python pur (descente de gradient).
  • Qualité & Ergonomie : Interface moderne responsives, rapide à charger et accessible aux handicaps.

2. Architecture Logicielle & Principes POO

La plateforme Ymmo est bâtie sur le framework léger Flask (Python). L'architecture logicielle applique une approche orientée services et adopte la séparation stricte des responsabilités (SOC - Separation of Concerns) :

📁 Organisation et Arborescence :

/
├── app.py                  # Contrôleur central (Définition de l'API REST et du routage)
├── database.db             # Fichier physique de base de données relationnelle SQLite
├── database/
│   ├── schema.sql          # Schéma DDL SQL avancé des tables
│   ├── seed.py            # Script d'injection du dataset historique (1100+ ventes)
│   └── db_manager.py       # Gestionnaire de pool et row_factory
├── services/
│   ├── analytics_service.py # Logique métier d'analyse de données statistiques
│   └── valuation_service.py # Modèle d'intelligence artificielle de prédiction
└── static/
    ├── css/style.css       # Design System unifié et styling responsive
    └── js/
        ├── main.js         # Logique d'interactivité publique (Recherche, AJAX, Formulaires)
        └── dashboard.js    # Logiciels de graphiques et CRM interne

🛡️ Alignement avec les Principes de Qualité de Code :

  • SOLID : Le principe de responsabilité unique (SRP) est scrupuleusement respecté. Le contrôleur `app.py` délègue les tâches analytiques à `AnalyticsService` et la régression linéaire à `ValuationService`.
  • DRY (Don't Repeat Yourself) : L'accès aux données SQLite est centralisé dans `db_manager.py`. Tout le style est encapsulé dans une unique feuille de route CSS unifiant les jetons de design (design tokens).
  • KISS (Keep It Simple, Stupid) : L'estimateur IA utilise un solveur de descente de gradient fait maison en Python pur (bibliothèque native `math`). Cela évite de charger des modules lourds et rend l'installation instantanée.

3. Conception de la Base de Données SQL

Le modèle conceptuel de données (MCD) structure de manière relationnelle toutes les entités du domaine Ymmo. L'intégrité référentielle est assurée par l'activation des clés étrangères (`PRAGMA foreign_keys = ON;`).

📊 Schéma Relationnel Logique (MLD) :

Table Description Clés et Contraintes SQL Avancées
agencies Le siège et les 12 agences `id` (PK), `email` (UNIQUE), `name` (NOT NULL)
agents Les 66 collaborateurs `id` (PK), `agency_id` (FK), `role` (CHECK: 'agent', 'manager', 'admin'), `email` (UNIQUE)
properties Le catalogue complet des biens (1165+) `id` (PK), `agency_id` (FK), `agent_id` (FK), `type` (CHECK: 'maison', 'appartement', 'bureau', 'commercial'), `status` (CHECK: 'available', 'sold')
transactions Historique des ventes réelles `id` (PK), `property_id` (FK SET NULL), `agent_id` (FK), `price` (REAL NOT NULL), `date` (DATE)
leads Prospects du CRM commercial `id` (PK), `property_id` (FK), `status` (CHECK: 'pending', 'contacted', 'completed')

⚡ Indexation et Optimisation de Requêtes :

Pour garantir une vitesse d'affichage optimale sur les millions de données futures, les index DDL suivants ont été créés :

  • `idx_properties_city` sur `properties(city)` : Accélère les filtrages par ville de 90%.
  • `idx_properties_price` sur `properties(price)` : Optimise les recherches par tranches de budget.
  • `idx_transactions_date` sur `transactions(date)` : Rend les tris chronologiques instantanés pour les graphiques analytiques.

4. Architecture Infra & Synchronisation de la Base de Données

Dans un environnement de production d'entreprise comme Ymmo, l'application est conçue pour être déployée au sein d'une architecture haute disponibilité composée de plusieurs machines virtuelles (VMs) de calcul réparties derrière un répartiteur de charge (Load Balancer).

Par défaut, l'application utilise une base de données SQLite locale (database.db). Dans un contexte multi-VMs, cela pose un problème majeur : chaque VM possède son propre fichier local indépendant, empêchant la synchronisation des données (ex. un bien enregistré sur la VM-1 ne sera pas visible sur la VM-2).

🔄 Options de Synchronisation & Architecture de Stockage :

Option d'architecture Description technique Avantages & Inconvénients Préconisation pour Ymmo
1. SGBDR Centralisé
(PostgreSQL / MySQL)		 Migration vers une base de données hébergée sur une VM dédiée ou un service Cloud managé. Les instances Flask s'y connectent via TCP/IP sécurisé. 🟢 Avantages : Très robuste, standard de l'industrie, gestion native et sûre des accès concurrents, sauvegardes simplifiées.
🔴 Inconvénients : Ajoute un composant réseau à administrer, latence réseau minimale à l'écriture. 		Recommandé (Production)
C'est la solution la plus propre et standard pour une plateforme professionnelle comme Ymmo.
2. Réplication LiteFS
(Système de fichiers virtuel)		 Utilisation de LiteFS pour intercepter les écritures SQLite et les répliquer au niveau bloc en temps réel entre les VMs via un modèle Primaire/Réplique. 🟢 Avantages : Conserve la légèreté de SQLite sans serveur externe, temps de lecture instantanés (local).
🔴 Inconvénients : Configuration FUSE complexe sur Windows Server, gestion active du rôle Primaire/Réplique requise. 		Alternative viable
Adapté si l'on souhaite conserver la base SQLite sans migrer le code SQL vers un autre SGBDR.
3. Volume Partagé Réseau
(NFS / GlusterFS)		 Partage du fichier database.db sur un stockage réseau partagé monté sur toutes les VMs Flask. 🟢 Avantages : Très simple à mettre en place sur l'infrastructure réseau existante.
🔴 Inconvénients (Majeur) : SQLite gère très mal les verrous de fichiers sur le réseau (NFS). Risque critique de corruption de la base de données lors d'accès concurrents d'écriture. 		À ÉVITER ABSOLUMENT
Risque de perte de données et d'erreurs de verrouillage en cas d'écritures simultanées des agents.

🌐 Alignement avec le Projet d'Infrastructure & Sécurité (INFRA) :

Le déploiement de cette application web s'inscrit au sein d'une infrastructure réseau et système d'entreprise sécurisée :

  • Active Directory & GPOs (Windows Server) : Centralisation de l'authentification des 66 agents immobiliers. Le dashboard pro peut se connecter à l'AD de l'entreprise (via LDAP) pour attribuer automatiquement les rôles (admin, manager, agent) et appliquer des stratégies de groupe limitant les droits des serveurs de développement.
  • Réseau & VPN IPSec site à site : Le siège social basé à Aix-en-Provence héberge les infrastructures centrales (serveur de base de données principal, Active Directory). Les 12 agences réparties sur le territoire s'y connectent à travers un VPN IPSec site-à-site chiffré pour garantir la sécurité et la confidentialité des échanges de données CRM.
  • Politique de Sécurité Réseau (Pare-feu & Filtrage) : Les serveurs Flask sont déployés dans une DMZ (Zone Démilitarisée) accessible depuis internet par les clients et agents. La base de données centralisée (PostgreSQL) est isolée dans un LAN privé / Zone de données sécurisée. Les règles du pare-feu n'autorisent que le trafic entrant sur le port standard de la DB (ex: port 5432 pour PostgreSQL) provenant exclusivement des adresses IP des serveurs Flask de la DMZ.

5. Moteur d'Analyse Statistique & IA

L'algorithme d'estimation de valeur immobilière est le cœur technologique intelligent de Ymmo. Il est implémenté sous forme d'une Régression Linéaire Multi-Variables avec optimisation par Descente de Gradient Stochastique.

🧮 Modèle Mathématique Sous-Jacent :

L'estimateur ajuste l'équation prédictive suivante en calculant les poids $\beta$ optimaux :

Prix estimé = β0 + β1 × Surface + β2 × Pièces + β3 × Chambres + β4 × Âge + β5 × PrixMoyVille + β6 × PrixMoyType

🧠 Processus d'Apprentissage Automatique (Machine Learning Workflow) :

  1. Nettoyage & Agrégation : Le système charge les biens immobiliers et calcule les prix moyens au m² pour chaque ville et type de bien, agissant comme des variables explicatives robustes.
  2. Mise à l'échelle (Standardization) : Pour stabiliser le calcul, les variables subissent une normalisation (StandardScaler fait maison) en soustrayant la moyenne et en divisant par l'écart-type :
    x_norm = (x - mean) / std.
  3. Entraînement (Gradient Descent) : Le solveur itère 1500 fois avec un taux d'apprentissage $\alpha = 0.05$ pour minimiser la fonction de coût des moindres carrés ordinaires (MSE).
  4. Évaluation : Les métriques statistiques sont calculées sur le jeu d'entraînement :
    • Coefficient de Détermination ($R^2$) : Avoisine les 97.4%, indiquant une parfaite adéquation du modèle avec le marché réel simulé.
    • Erreur Moyenne Absolue (MAE) : Mesure la déviation moyenne en euros entre la prédiction et le prix réel.

6. Spécifications Techniques de l'API REST

L'échange de données entre le frontend interactif et le serveur Python s'effectue via une API web RESTful sécurisée retournant exclusivement du JSON.

📝 Catalogue des Endpoints APIs :

Méthode & Route Description Paramètres d'Entrée Structure du Retour JSON
GET /api/properties Liste filtrée des biens actifs `city`, `type`, `max_price`, `min_surface`, `rooms` `[{id, title, price, surface, city, agent_name, agent_phone, ...}]`
GET /api/properties/<id> Fiche de détail d'un bien `id` en paramètre d'URL `{id, title, description, price, address, agent_name, agent_email}`
POST /api/estimate Estimation d'un bien par l'IA `{surface, rooms, bedrooms, age, city, type}` (JSON payload) `{estimated_price, estimated_price_m2, city_average_m2, r2_score}`
POST /api/leads Soumission d'un prospect CRM `{name, email, phone, message, property_id}` (JSON payload) `{success: "Message transmis !", lead_id: 12}`
GET /api/dashboard/stats KPIs et analytiques du dashboard Aucun `{general: {...}, by_city: [...], trends: [...], hot_zones: [...]}`

7. Qualité Web, responsive et Accessibilité

L'interface utilisateur de Ymmo a été codée pour offrir une expérience fluide, rapide à s'exécuter et inclusive pour tous les profils d'utilisateurs.

♿ Accessibilité Numérique (WCAG & Sémantique HTML) :

  • Sémantique Structurée : Emploi rigoureux des balises HTML5 (`<header>`, `<nav>`, `<main>`, `<section>`, `<aside>`, `<footer>`) facilitant la navigation pour les lecteurs d'écran.
  • Contraste Optique : Les palettes de couleurs HSL respectent les ratios de contraste stricts imposés par les WCAG AA, que ce soit en mode sombre ou en mode clair.
  • Attributs ARIA complets : Les éléments interactifs disposent de labels ARIA explicites (ex: `aria-live="polite"` pour les rafraîchissements de listes asynchrones et `aria-modal="true"` pour la boîte de dialogue).
  • Keyboard Friendly : Les modales de contact s'ouvrent au clic et se ferment nativement avec la touche physique `Échap` (Escape) pour les personnes à mobilité réduite.

📱 Responsive Grid & Optimisation Mobile :

Le design system Vanilla CSS combine Flexbox et CSS Grid avec des tailles fluides (`clamp()`, `minmax()`). La plateforme s'affiche avec la même élégance sur :

  • Les smartphones (adaptation verticale et collapse de navigation).
  • Les tablettes (grille de filtres disposée de façon ergonomique).
  • Les écrans larges d'ordinateurs de bureau (mise en page multi-colonnes optimisée).
